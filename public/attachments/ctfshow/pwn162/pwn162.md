  - 你在第一阶段之前拿不到 libc 基址。
  - 你又只能做部分覆写，没法一次把完整目标地址写准。

  如果要“精确”，必须满足至少一个条件：

  - 先找到别的 info leak，先泄 libc。
  - 或者有更强的任意写，能直接写完整地址。
  - 或者能在同一进程里多次试错而不崩，但这题错一次通常就把进程打死了。

  所以这题最合理的做法就是爆破那 16 个候选值。理论期望大约 8 次命中一次，最坏 16 次。
  这也是为什么我把脚本改成自动重连重试，而不是手工碰运气。

• 先纠正一个点：

  不是“程序只给了你写 2 字节的原语”，而是“在第一次泄露 libc 之前，你只敢精确地改 2 字节”。
  真正的输入函数 read 其实能写很多字节，证据就在 C:/Users/34831/Desktop/pwn/Code/python/堆溢出/ctfshow堆溢出/pwn162 的 add 逻辑里：

  - 0xbcb: malloc(0x28)，分配 note 结构体
  - 0xc4d: malloc(size)，分配 name 缓冲区
  - 0xc90: read(0, name, size)，把你输入的 name 写进去
  - 0xc9d: mov [note+0x8], name
  - 0xcc4: scanf("%23s", note+0x10)，把 message 写到 note+0x10

  这说明程序本身是能让你写很多字节的。
  你在 exp.py:97 里只写 p16(...)，是因为这是第一次泄露前最稳的做法。

  1. 这个题里的 note 结构长什么样
  从反汇编可以直接还原成近似这样：

  struct Note {
      int used;          // +0x00
      int pad;           // +0x04
      char *name;        // +0x08
      char message[0x18];// +0x10
  };

  依据就是：

  - 0xc9d 把 malloc(size) 返回值写到 [note+0x8]
  - 0xcc4 把 message 读到 note+0x10
  - 0xccd 把 [note] = 1

  所以 exp.py:103 里每次 add()，实际上做了两次堆分配：

  - 一次给 Note 本体
  - 一次给 name

  2. 漏洞点在哪里
  delete 的核心在 C:/Users/34831/Desktop/pwn/Code/python/堆溢出/ctfshow堆溢出/pwn162 的这些位置：

  - 0xe90: mov dword ptr [note], 0
  - 0xeae: 取出 note->name
  - 0xeb5: free(note->name)

  问题是它只 free(name)，但没有：

  - free(note)
  - 清空 note->name
  - 清空全局数组里的这个 note 指针

  所以删完以后，程序还握着一个“悬空指针”。
  这就导致：

  - 同一个 index 可以再次 delete，形成 double free
  - 也可以继续通过旧 note 间接影响已经 free 的 chunk

  这就是整题的根。

  3. 堆基础：free 后 chunk 里会被写什么
  你可以把 glibc 的 chunk 想成：

  [ chunk header ][ user data ]

  malloc 返回给你的，是 user data 开头。
  free 以后，glibc 会把这块 user data 当成链表节点来用。

  不同 bin 里会写不同东西：

  - fastbin/tcache: 前面会放单链表指针 fd
  - unsorted bin: 前面会放双链表指针 fd 和 bk

  而这些 fd/bk 往往是 libc 里的地址，比如 main_arena 附近。

  这就是为什么 unsorted bin 经常能用来“借 libc 地址”。

  4. 为什么这里是“只改低 2 字节”
  关键思路是：

  - 先 free 一个大一点的 chunk，让它进 unsorted bin
  - 这时 chunk 头部附近已经被 glibc 写入了一个真实的 libc 指针
  - 这个指针的高 6 个字节已经是对的，因为它本来就指向 libc
  - 你不知道 libc 基址，所以不可能凭空写出完整 8 字节目标地址
  - 但你可以“借壳”：保留高位，只修改低位，让它从 main_arena+某偏移 改成 _IO_2_1_stdout_ - 0x43

  也就是：

  原来: 0x7f xx xx xx xx xx ?? ??
  目标: 0x7f xx xx xx xx xx 55 dd

  高 6 字节不用你猜，因为它们已经在那个 unsorted bin 指针里了。
  你只需要改最后 2 字节。

  这就是 exp.py:97 里：

  stdout_partial = p16(stdout_low)

  的本质。

  5. 那为什么还要爆破 16 次
  因为 ASLR 下，libc 基址按页对齐。

  这意味着：

  - 最低 12 bit 固定
  - _IO_2_1_stdout_ - 0x43 的低 12 bit 恒定是 0x5dd
  - 但低 16 bit 里的高 4 bit 会变化

  所以候选只有 16 个：

  - 0x05dd
  - 0x15dd
  - ...
  - 0xf5dd

  这就是 exp.py:46 枚举 ?5dd 的原因。

  6. 为什么偏偏是 stdout - 0x43
  这里要补一点 fastbin attack 的基本规律：

  你伪造的是“chunk 地址”，但 malloc 返回的是“user data 地址”，也就是：

  returned_ptr = fake_chunk_addr + 0x10

  而我们想让最终拿到的可写区域正好落在 stdout 前面一点，这样就能先写 padding，再写 _flags 等字段。

  所以选：

  fake_chunk = _IO_2_1_stdout_ - 0x43
  returned_ptr = _IO_2_1_stdout_ - 0x33

  b"\x00" * 0x33 + p64(0xFBAD1800) + ...

  前面 0x33 字节是对齐填充，后面正好覆盖到 stdout 的关键字段。

  7. _IO_FILE / stdout 是什么
  stdout 不是一个简单整数，它是 libc 里的一个大结构体，大致像这样：

  struct _IO_FILE {
      int _flags;
      char *_IO_read_ptr;
      char *_IO_read_end;
      char *_IO_read_base;
      char *_IO_write_base;
      char *_IO_write_ptr;
      char *_IO_write_end;
      char *_IO_buf_base;
      char *_IO_buf_end;
      ...
  };

  你不需要一下把它学全。
  这题只要知道两件事：

  - stdout 是 libc 里的全局对象 _IO_2_1_stdout_
  - 如果把它某些字段伪造成特殊状态，下一次程序输出时，libc 会把内部指针“漏出来”

  所以 exp.py:77 的 fake_stdout，本质是在伪造这个结构，让 puts/printf 之类的输出把 libc 地址带出来。

  8. 这题的堆链，按最基础的话怎么理解
  可以粗糙理解成 4 步：

  9. 先造出一个 unsorted bin chunk
     它里面天然带 libc 指针
  10. 利用 double free 做 fastbin dup
     让 malloc 将来返回到你想去的地方
  11. 第一次只做“部分指针覆盖”
     把原本的 libc 指针，低 2 字节改成 stdout-0x43 附近
  12. 拿到 stdout 泄露后
     libc 基址就精确了，第二次再精确打 __malloc_hook

  所以“不精确”只发生在第一跳。
  第一跳过去以后，后面就是精确 exp。

› 为什么要把末两位改成dd？io的位置是固定的还是什么，dd有什么说法吗


• dd 不是玄学，它是算出来的。

  先看目标地址不是 stdout 本身，而是：

  _IO_2_1_stdout_ - 0x43

  这题用的 libc-2.23.so 里，_IO_2_1_stdout_ 的偏移是：

  0x3c5620

  所以目标偏移就是：

  0x3c5620 - 0x43 = 0x3c55dd

  你看到的 dd 就是这么来的。
  更直观点看最后 2 字节：

  0x5620 - 0x0043 = 0x55dd

  所以不是“末两位随便改成 dd”，而是“目标地址的最低字节本来就是 0xdd”。

  为什么 stdout 的位置像是固定的？

  是“相对 libc 固定”，不是“绝对地址固定”。

  也就是说：

  - _IO_2_1_stdout_ 在这份 libc-2.23.so 里的偏移固定是 0x3c5620
  - 但 libc 整体会被 ASLR 搬到不同基址
  - 所以真实地址每次变，但“libc_base + 0x3c5620”这个关系不变

  比如某次 libc 基址是：

  0x7fde6ddca000

  那目标就是：

  0x7fde6ddca000 + 0x3c55dd = 0x7fde6e18f5dd

  你看，最后还是 ...f5dd。
  另一次基址不同，就可能变成 ...25dd、...55dd，但最后 3 个十六进制位总是 5dd。

  为什么会变成 ?5dd 而不是固定 55dd？

  因为：

  - libc 基址按 0x1000 页对齐
  - 所以最低 12 bit 固定为 0
  - 目标偏移 0x3c55dd 的最低 12 bit 是 0x5dd
  - 因而真实地址最低 12 bit 永远是 5dd
  - 但再往上的那 4 bit 会跟着 ASLR 变，所以是 ?5dd

  这就是为什么要爆破：

  - 0x05dd
  - 0x15dd
  - ...
  - 0xf5dd

  在脚本里就是这里在做这件事：
  exp.py:46
  exp.py:97

  为什么你看到的是 \xdd\x55？

  因为 x86-64 是小端序。

  数值 0x55dd 在内存里写进去是：

  \xdd\x55

  最低字节先放，所以你总先看到 dd。

  一句话总结

  - dd 来自 _IO_2_1_stdout_ - 0x43
  - stdout 不是绝对地址固定，而是“在 libc 里的偏移固定”
  - ASLR 只会让它变成 ?5dd
  - 所以第一阶段才要枚举 16 个低 16 位候选值
